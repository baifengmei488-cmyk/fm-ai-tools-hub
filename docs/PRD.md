# ToolVault 产品需求文档（PRD/RPD）

本文按 PRD（Product Requirements Document，产品需求文档）组织；如果项目内部习惯称为 RPD，也可以把本文作为同一份需求基线使用。

## 1. 文档目的

本文用于说明 ToolVault 的产品背景、目标用户、核心场景、功能范围、内容安全要求和验收标准。它不是实施计划，也不替代技术架构文档；具体技术分层见 `docs/ARCHITECTURE.md`。

## 2. 项目背景

ToolVault / FM AI Tools Hub 是一个 local-first 的 AI 工具知识库项目。项目要解决的问题不是“把工具列出来”这么简单，而是把 Claude Code、MCP、plugins、skills、CLI、测试工具和日常开发经验整理成可浏览、可检索、可更新、可验证的知识产品。

当前工具资料通常分散在命令行、官方文档、本地配置、对话经验和临时笔记里。使用者经常遇到以下问题：

- 知道装了某个工具，但不知道适合在什么场景使用。
- 有安装命令，但缺少验证方式和安全边界。
- 有工具说明，但缺少组合工作流、提示词模板和实际验收方式。
- 更新后页面内容可能被短样例覆盖，导致历史内容丢失。
- 导入记录不够清晰时，很难追踪“哪些页面、哪些工具、哪些字段发生过变化”。

ToolVault 的目标是把这些经验沉淀为稳定、可复核、可持续更新的工具知识库。

## 3. 目标用户

### 主要用户

- FM 自己：用于管理 AI 工具、测试工具、开发工具、工作流和项目经验。
- 使用 FM AI Tools Hub 的访问者：浏览公开工具库、指南、工作流、提示词和命令速查。
- 项目维护者：通过后台导入经过审阅的 JSON，更新工具内容和页面内容。

### 次要用户

- 需要了解项目能力的协作者或面试评估者。
- 后续接手维护的人：通过 README、PRD、架构文档和更新日志理解项目边界。

## 4. 产品目标

1. 建立一个可公开浏览的工具知识库。
2. 支持工具详情页展示安装、验证、使用指南、分类和标签。
3. 支持工作流、提示词、命令速查和指南导航，让工具从“单点能力”变成“可执行流程”。
4. 支持导入历史和更新日志，记录每次导入影响了哪些页面、工具和字段。
5. 支持公开/登录可见性，避免未准备公开的内容泄露。
6. 在导入前进行敏感内容扫描，避免真实密钥、token、cookie、私钥或私有内容进入数据库。
7. 通过 E2E 测试保护主要页面和历史内容不被清空。

## 5. 非目标

首版及当前阶段不做以下内容：

- 多用户注册和复杂权限体系。
- 评论、点赞、收藏等社区功能。
- Web 服务直接扫描本机 Claude 配置、MCP 设置或系统目录。
- 自动抓取付费、登录态或内部资料。
- 自动发布 skill、plugin 或外部包。
- 存储真实生产密钥、私有 URL、客户数据或内部机密内容。
- 用 ToolVault 执行导入 payload 里的命令。

## 6. 核心使用场景

### 场景 1：公开访问者浏览工具库

访问者进入工具库，可以看到公开工具卡片、类型、状态、简要说明和图标。点击工具后进入详情页，查看工具速查、安装与验证、使用指南和相关导航。

验收重点：公开用户只能看到 `public` 内容，不能看到 `login_required` 工具或指南。

### 场景 2：访问者查看工作流和提示词

访问者进入工作流页面，可以在工作流、提示词、命令三个栏目之间切换，理解不同工具如何组合使用。

验收重点：`/prompts` 和 `/commands` 旧入口应重定向到 `/workflows?tab=prompts` 和 `/workflows?tab=commands`，避免旧链接失效。

### 场景 3：维护者导入经过审阅的 JSON

维护者先由 Claude 生成 reviewed JSON，再在后台或 CLI 中导入。后端校验 schema、扫描敏感内容、写入工具、指南、页面内容、来源、变更和更新日志。

验收重点：导入成功后要能看到 created/updated/deleted 数量、content_plan、change_details 和 update log 记录。

### 场景 4：维护者追踪更新日志

访问者或维护者进入更新日志页面，可以看到每次导入的结果、验证结论、影响工具、具体更新内容和来源。

验收重点：更新日志不能暴露私有内容；字段级变更应尽量指向具体 page_path、section、tool_slug 和 field。

### 场景 5：访问者浏览 About FM 页面

访问者进入 `/about`，看到独立风格的个人作品集页面，理解 FM 的业务理解、测试负责人经验、跨部门沟通、自动化能力、AI 工具整合能力和项目沉淀方式。

验收重点：页面不展示真实姓名、电话，也不出现直接求职表述或简历下载入口；只展示已确认的邮箱和微信号。

## 7. 功能范围

### 7.1 公开工具库

- 展示公开工具列表。
- 支持按工具类型筛选。
- 展示工具名称、摘要、状态、类型、分类等信息。
- 不展示未公开工具。

### 7.2 工具详情与指南

- 展示工具基本信息、安装命令、验证命令。
- 渲染 Markdown 使用指南。
- 使用安全的 Markdown 渲染策略，避免不可信 HTML 执行。
- 支持指南目录和详情页内导航。

### 7.3 工作流、提示词和命令速查

- `/workflows` 展示工具组合工作流。
- 同一页面通过 tab 展示提示词模板和命令汇总。
- `/prompts`、`/commands` 作为旧入口重定向。
- 页面内容来自导入 payload 的 `page_content`。

### 7.4 更新日志

- 展示每次导入的来源、验证结论、影响范围和结果摘要。
- 支持展开查看执行详情和具体更新内容。
- 具体更新内容应包含页面、字段、工具和说明。

### 7.5 后台登录与导入

- 使用单管理员登录。
- 登录后可访问后台工具列表和 JSON 导入页。
- 导入失败时显示明确错误，不离开当前页面。
- 登录态只用于后台和 login_required 内容，不影响公开页面浏览。

### 7.6 About FM 个人主页

- 独立视觉风格，不强制与工具库页面完全一致。
- 内容强调业务理解、测试质量、流程拆解、自动化落地、产品化沉淀和 AI 工具协作。
- 不使用“求职”“找工作”“投递”等直接表达。
- 不提供电话、真实姓名或简历下载。

## 8. 内容与安全要求

- 导入内容只允许公开资料、官方资料或明确授权资料。
- 不写入真实 API key、token、cookie、密码、私钥、生产连接串、私有 URL 或内部资料。
- 对于没有安全公开来源的新工具，可以生成明确标注的 baseline guide，但不能留空详情页。
- 保留仍正确的旧内容，只更新新增能力、过时步骤、错误描述、缺失场景、具体 prompts/commands、组合方式、安全边界或验证细节。
- 页面内容不得因为短样例、测试 payload 或局部导入而清空完整历史内容。

## 9. 数据导入要求

导入 payload 应包含：

- `tools`：工具主体和指南。
- `page_content`：首页、工作流、提示词、命令、指南等页面内容。
- `content_plan`：页面路径和栏目级刷新清单。
- `sources`：资料来源。
- `changes`：更新批次摘要。
- `change_details`：字段级变更记录。

导入流程要求：

1. 先校验 JSON 格式和 schema。
2. 再执行敏感内容扫描。
3. 再按 slug upsert 工具、指南、分类、标签和页面内容。
4. 记录 import batch 和 update log。
5. 返回 created、updated、deleted、content_plan_items、change_details、page_content_sections 等统计信息。

## 10. 验收标准

### 公开页面

- 首页可正常展示推荐内容。
- `/tools` 可展示公开工具并支持类型筛选。
- `/tools/{slug}` 可展示工具详情、安装验证和使用指南。
- `/workflows` 可在工作流、提示词、命令之间切换。
- `/guides` 可展示工具使用导航和安全注意事项。
- `/updates` 可展示导入日志和字段级更新内容。
- `/about` 可展示个人主页内容，并满足隐私表达要求。

### 后台导入

- 管理员可登录。
- 合法 JSON 可导入成功。
- 非法 JSON 显示错误提示。
- login_required 工具公开不可见，后台可见。

### 内容保护

- `/api/tools` 返回非空工具列表。
- `/api/page-content` 的核心 section 返回非空数组。
- `/api/update-logs` 返回非空更新日志。
- 新功能变更后，不能把历史工具内容、页面内容或更新日志清空。

### 安全

- 导入 payload 中明显真实的敏感信息应被拒绝。
- 后端不执行导入 payload 中的 command 字段。
- 文档、fixtures、截图和示例中不出现真实密钥或生产连接信息。

## 11. 后续迭代方向

- 为更多工具补充高质量指南、组合工作流和验证模板。
- 改善 update log 的字段级差异展示。
- 增加更细的内容质量检查，防止短内容覆盖长内容。
- 为 PRD、架构文档和导入规范建立固定维护流程。
- 将稳定流程沉淀为可复用 skill 或 runbook。
- 根据部署需要补充生产环境配置、备份和监控方案。

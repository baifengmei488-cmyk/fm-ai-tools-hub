# ToolVault E2E Testing Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a repeatable Playwright-based E2E testing workflow for ToolVault, record known runtime fixes, and make it easy to automatically run the real app flow.

**Architecture:** Keep backend unit/API tests in `backend/tests`, add browser E2E tests under `frontend/e2e`, and add one root script that prepares a temporary SQLite runtime environment, starts FastAPI and Vite, runs Playwright, and cleans up. The E2E test drives the app through the same surfaces a user uses: browser pages, forms, navigation, and backend HTTP through the frontend.

**Tech Stack:** FastAPI, Alembic, SQLite test database, React/Vite, Playwright test runner, Bash, pytest, ruff, npm scripts.

---

## File Structure

Create and modify these files:

```text
toolvault/
  docs/testing/e2e-fixes.md             # Human-readable record of runtime issues and fixes
  scripts/run-e2e.sh                    # One-command E2E environment launcher and test runner
  frontend/package.json                 # Add Playwright scripts and dev dependency
  frontend/package-lock.json            # Updated by npm install
  frontend/playwright.config.ts         # Playwright config for local E2E execution
  frontend/e2e/toolvault.spec.ts        # Browser E2E scenarios
```

Responsibilities:

- `scripts/run-e2e.sh`: owns runtime orchestration only. It creates the temp DB, migrates, imports seed data, starts/stops services, and runs Playwright.
- `frontend/playwright.config.ts`: owns Playwright test settings only. It uses Chromium and a fixed base URL.
- `frontend/e2e/toolvault.spec.ts`: owns user-facing browser behavior tests only. It should not import internal React modules or backend internals.
- `docs/testing/e2e-fixes.md`: records observed runtime issues and fixes. It is documentation, not a test spec.

- [ ] **Step 1: Write the runner script**

Create `scripts/run-e2e.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
BACKEND_PORT="${TOOLVAULT_E2E_BACKEND_PORT:-8120}"
FRONTEND_PORT="${TOOLVAULT_E2E_FRONTEND_PORT:-5173}"
DB_PATH="${TOOLVAULT_E2E_DB_PATH:-$(mktemp -t toolvault-e2e-XXXXXX.db)}"
BACKEND_LOG="$(mktemp -t toolvault-backend-XXXXXX.log)"
FRONTEND_LOG="$(mktemp -t toolvault-frontend-XXXXXX.log)"
BACKEND_PID=""
FRONTEND_PID=""

cleanup() {
  if [[ -n "$FRONTEND_PID" ]]; then
    kill "$FRONTEND_PID" 2>/dev/null || true
  fi
  if [[ -n "$BACKEND_PID" ]]; then
    kill "$BACKEND_PID" 2>/dev/null || true
  fi
  rm -f "$DB_PATH" "$BACKEND_LOG" "$FRONTEND_LOG"
}
trap cleanup EXIT

require_free_port() {
  local port="$1"
  if lsof -nP -iTCP:"$port" -sTCP:LISTEN >/dev/null 2>&1; then
    echo "Port $port is already in use." >&2
    return 1
  fi
}

wait_for_url() {
  local url="$1"
  local name="$2"
  for _ in $(seq 1 60); do
    if curl -fsS "$url" >/dev/null 2>&1; then
      return 0
    fi
    sleep 0.5
  done
  echo "$name did not become ready at $url" >&2
  echo "Backend log:" >&2
  cat "$BACKEND_LOG" >&2 || true
  echo "Frontend log:" >&2
  cat "$FRONTEND_LOG" >&2 || true
  return 1
}

require_free_port "$BACKEND_PORT"
require_free_port "$FRONTEND_PORT"

export TOOLVAULT_DATABASE_URL="sqlite+pysqlite:///$DB_PATH"
export TOOLVAULT_CORS_ORIGINS="http://localhost:$FRONTEND_PORT,http://127.0.0.1:$FRONTEND_PORT"

cd "$ROOT_DIR/backend"
uv run alembic upgrade head
uv run python -m app.cli.import_tools ../fixtures/sample-tool-import.json >/dev/null
uv run uvicorn app.main:app --host 127.0.0.1 --port "$BACKEND_PORT" >"$BACKEND_LOG" 2>&1 &
BACKEND_PID="$!"

wait_for_url "http://127.0.0.1:$BACKEND_PORT/api/health" "Backend"

cd "$ROOT_DIR/frontend"
VITE_API_BASE="http://127.0.0.1:$BACKEND_PORT" npm run dev -- --host 127.0.0.1 --port "$FRONTEND_PORT" >"$FRONTEND_LOG" 2>&1 &
FRONTEND_PID="$!"

wait_for_url "http://127.0.0.1:$FRONTEND_PORT" "Frontend"

PLAYWRIGHT_BASE_URL="http://127.0.0.1:$FRONTEND_PORT" npm run e2e
```

```markdown
# ToolVault E2E Fix Log

## 2026-06-18 - Loopback frontend origin was blocked by CORS

- 触发方式：启动 Vite 时使用 `--host 127.0.0.1`，浏览器从 `http://127.0.0.1:5173` 请求 FastAPI API。
- 观察结果：公开工具库页面显示“工具加载失败，请稍后重试。”，浏览器控制台报 CORS 错误，API 响应缺少 `Access-Control-Allow-Origin`。
- 根因：后端默认 CORS 配置只包含 `http://localhost:5173`，不包含计划中使用的 `http://127.0.0.1:5173`。
- 修复：将默认 `TOOLVAULT_CORS_ORIGINS` 和 `.env.example` 更新为同时包含 `localhost` 与 `127.0.0.1` 前端来源，并新增配置测试覆盖默认来源。
- 验证：`uv run --directory backend pytest tests/test_config.py -v` 通过；运行时用浏览器打开 `http://127.0.0.1:5173/tools` 可加载 `Playwright MCP`。
```

- [ ] **Step 4: If E2E finds a bug, follow this exact fix loop**

For each bug discovered, do not patch blindly. Use `superpowers:systematic-debugging` and `superpowers:test-driven-development`.

1. Capture the failing Playwright output.
2. Identify root cause by checking browser output, network requests, backend logs, and recent code.
3. Add or adjust the smallest failing test that reproduces the bug. Prefer E2E if the issue is cross-stack; use pytest if the issue is backend-only.
4. Run that test and confirm it fails for the expected reason.
5. Implement the minimal fix.
6. Re-run the failing test and confirm it passes.
7. Re-run `./scripts/run-e2e.sh`.
8. Append a record to `docs/testing/e2e-fixes.md` using the existing format.
9. Commit the fix with a message that starts with `fix:`.
